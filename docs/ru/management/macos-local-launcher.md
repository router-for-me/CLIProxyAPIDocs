# Локальный лаунчер для macOS

Эта страница описывает практичную схему запуска CLIProxyAPI на macOS:

- основной checkout хранится в постоянном каталоге с правом записи
- в `/Applications` лежит тонкий `.app`-лаунчер
- после старта автоматически открывается встроенный Web UI
- обновление выполняется отдельным процессом

Идея в том, чтобы разделить состояние рантайма, обновления и сам `.app` bundle.

## Рекомендуемая структура каталогов

Храните реальный checkout вне `.app`, например:

```text
~/CLIProxyAPI/
  bin/
  auths/
  logs/
  temp/
  config.yaml
```

Назначение каталогов:

- `config.yaml`: локальная конфигурация
- `auths/`: OAuth и provider auth files
- `logs/`: runtime-логи
- `bin/`: собранные бинарники и вспомогательные скрипты
- `temp/`: PID-файлы, профили браузера и временные управляющие файлы

## Делайте `.app` тонким лаунчером

На macOS `.app` лучше использовать как тонкий лаунчер, а не как вторую установку сервера.

Рекомендуемый шаблон:

1. Держите реальный checkout в стабильном каталоге, например `~/CLIProxyAPI`
2. Поместите небольшой `.app` в `/Applications`
3. Пусть `.app` вызывает скрипты из реального checkout

Так вы не встраиваете устаревающий бинарник в сам bundle.

## Поток запуска

Надёжный сценарий запуска обычно такой:

1. Проверить, не запущен ли CLIProxyAPI уже сейчас
2. Если нет, запустить локальный бинарник в detached-режиме
3. Дождаться ответа от `http://127.0.0.1:8317/`
4. Дождаться доступности `http://127.0.0.1:8317/management.html`
5. Открыть Web UI

Встроенная панель управления доступна по адресу:

```text
http://127.0.0.1:8317/management.html
```

Не используйте `/` как вход в management UI. Корневой путь предназначен только для лёгкой проверки состояния.

## Поток остановки

Если лаунчеру нужна кнопка остановки, удобно сохранять симметрию:

1. Остановить локальный процесс CLIProxyAPI через PID-файл или отдельный status-скрипт
2. Закрыть только то окно Web UI, которое относится к CLIProxyAPI

Не закрывайте произвольные вкладки из обычной браузерной сессии пользователя.

## Используйте отдельный профиль браузера для Web UI

Если нужен надёжный сценарий "закрыть Web UI", не открывайте панель в случайной уже существующей вкладке браузера.

Лучше использовать отдельный профиль или app-style window.

### Chrome или Chromium

```bash
open -na "Google Chrome" --args \
  --user-data-dir="$HOME/CLIProxyAPI/temp/webui-browser-profile" \
  --app="http://127.0.0.1:8317/management.html"
```

Обычно это самый удобный вариант, если потом нужно закрыть именно это окно Web UI.

### Firefox

```bash
open -na "Firefox" --args \
  --profile "$HOME/CLIProxyAPI/temp/webui-firefox-profile" \
  --new-window "http://127.0.0.1:8317/management.html"
```

Этот вариант тоже изолирует профиль, хотя выглядит менее похоже на отдельное desktop-приложение, чем Chrome `--app`.

### Safari

Safari может просто открыть URL:

```bash
open -a "Safari" "http://127.0.0.1:8317/management.html"
```

Но у Safari нет прямо эквивалентного CLI-сценария с отдельным профилем и app-style окном.

Если вам нужен детерминированный workflow "открыть только это окно" и "закрыть только это окно", обычно лучше подходят Chrome-подобные браузеры или Firefox. Если вы предпочитаете Safari, проверьте актуальную документацию Safari и macOS automation.

## Обновления должны быть отдельны от запуска

Не стоит перекладывать автоматическое обновление репозитория на `.app`.

Лучше держать обновление отдельным терминальным workflow, например:

- `up`
- `git pull`
- локальный скрипт пересборки

Так обновления остаются явными и их проще отлаживать.

## Совместно с Gemini CLI

Если вы используете Gemini CLI вместе с CLIProxyAPI на macOS, этот шаблон хорошо сочетается с настройкой из [Gemini CLI](/ru/agent-client/gemini-cli).

Обычное разделение ответственности:

- `.app`: запуск и остановка локального сервиса и Web UI
- terminal command: обновление checkout и пересборка бинарников
- `gemini`: подключение клиента к локальному endpoint CLIProxyAPI
